# Issue Writing

GitHub Issue を「読んだ人がすぐ作業に着手できる」品質で書くためのスキル。

## Template

### タイトル

```
<対象ファイル or コンポーネント>  <何が問題か / 何が必要か>
```

短く具体的に。ファイル名やモジュール名を先頭に置く。

### 本文構成

```markdown
## 基本方針
このIssueで目指す方向性。構造案（ディレクトリツリー）やコード例で具体的に示す。

### ワークフロー素案
作業手順のラフスケッチ。散文で流れを書く（箇条書きではなく口語調で）。

### 現状
現在のコード・設定をそのまま貼る（コードブロックで囲む）。
読み手が「何を変えるか」を判断できる材料。

## 参考
別リポジトリ・外部ドキュメントの関連ルールを `---` で区切って引用。
不要なら省略可。

## 作業上の注意
- [ ] 実装時の制約やプロセス指示をチェックボックスで記述
```

## Guidelines

1. **現状をまず見せる** — 変更前のコードや設定を省略しない。読み手が差分を判断できるようにする
2. **方針は具体的に** — 「改善する」ではなく、ディレクトリ構造やコード例で「こうしたい」を示す
3. **作業プロセスも指示** — 単に「直せ」ではなく「どう進めるか」も含める
4. **Topic to Article** — いきなりコードを書かず、`docs/<YYMMDD>-<topic>` に思考過程（Why / Which / 選択肢 / 結論）を記録してからコード変更と一緒にコミットする
5. **最小メタデータ** — ラベル・アサインは必要になるまで付けない

## Anti-Patterns

- ❌ タイトルが曖昧（「〇〇を改善」「バグ修正」）
- ❌ 現状コードの省略（「現在のコードに問題がある」だけ）
- ❌ 方針がポエム（「もっと良くしたい」— 具体案なし）
- ❌ 作業手順の丸投げ（「よしなにやって」）
- ❌ 1つの Issue に複数の無関係な変更を詰め込む
